
1. 可维护性
 
- 代码结构清晰，分层合理（MVC、DDD、微服务等）
- 命名规范、注释恰当、逻辑直白
- 重复代码少，遵循 DRY、单一职责原则
- 易于修改，改一处不牵一发而动全身
 
2. 可扩展性
 
- 模块解耦，依赖关系清晰
- 接口稳定，内部实现可替换
- 新增功能尽量不破坏原有逻辑
- 支持插件化、配置化扩展
 
3. 可测试性
 
- 逻辑可单元测试，无强耦合
- 有完善的自动化测试（单元、集成、E2E）
- 测试覆盖率合理，关键路径全覆盖
- 易于 Mock、Stub 外部依赖
 
4. 可构建性 & 可部署性
 
- 统一构建脚本（Maven/Gradle/npm/make）
- 环境隔离：开发/测试/生产配置分离
- 一键编译、打包、发布
- 支持容器化（Docker）、CI/CD 流水线
- 无“本地能跑，线上不行”的问题
 
5. 可靠性 & 健壮性
 
- 异常处理完善，不裸奔
- 边界条件、空值、并发场景考虑充分
- 有重试、降级、限流、熔断机制
- 日志规范、可追踪、可排查
 
6. 可观测性
 
- 统一日志格式
- 关键指标埋点（性能、调用量、错误率）
- 支持链路追踪（Trace）
- 方便监控告警
 
7. 规范性
 
- 统一代码风格（lint 强制校验）
- Git 分支规范、提交信息规范
- 代码评审流程
- 依赖版本统一管理
 
8. 安全性
 
- 输入校验、防注入、防 XSS、防 CSRF
- 敏感信息不硬编码，使用配置/密钥管理
- 依赖定期扫描，避免漏洞包
- 权限控制最小化
 
9. 性能友好
 
- 无明显内存泄漏、死循环、冗余计算
- 数据库索引、连接池、缓存设计合理
- 资源释放规范
- 支持压力测试与性能优化链路
 
10. 团队协作友好
 
- 文档齐全：架构、接口、部署、维护手册
- 新人易上手，构建/运行流程简单
- 冲突少、耦合低，多人并行开发不打架
 
11. 目录结构约束（auto_iteration）

- Python 执行源码统一放在 `src/`（如 `src/orchestrator_core/`、`src/scripts/`）
- 根目录仅保留入口与工程配置文件（如 `orchestrator.py`、`requirements.txt`、`pytest.ini`）
- 根目录 `scripts/` 仅放运维/封装脚本（`.sh`/`.ps1`），不再放 Python 业务源码
- 容器与部署编排统一放在 `deploy/`，例如 `deploy/docker/`
- 新增 Python 模块时，必须放在 `src/` 下并保持按职责分层（core / scripts / utils）
- 入口兼容要求：根目录入口（如 `orchestrator.py`）可作为稳定命令入口，但内部实现应委托 `src/`
- 路径变更后必须同步更新：README、部署手册、验收清单、CI 与 Docker 配置
- CI 校验建议：新增规则，禁止在根目录和 `scripts/` 新增 Python 业务源码（白名单入口文件除外）
